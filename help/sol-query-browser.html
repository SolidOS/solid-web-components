sol-query — Browser / CDN Guide

A complete guide to using <sol-query> in any web page without a build system. All dependencies are loaded from CDN; only the component files need to be served alongside your HTML.

Dependencies

The only external dependency is rdflib.js, loaded from CDN. The sol-query component files must be served locally (or from your own CDN).

Dependency CDN Notes
rdflib.js https://esm.sh/rdflib ES module; loaded via importmap so all modules share one instance
sol-query.js local Registers the <sol-query> custom element
shared/rdf-utils.js local (auto-imported) RDF engine; imported by sol-query.js automatically
utils/sol-query-ui.js local (auto-imported) Renderer; imported by sol-query.js automatically
utils/sol-query-mini.js local (auto-imported) Mini-query validator; imported by sol-query.js automatically
The importmap approach means you write "rdflib" as the import specifier and the browser resolves it to the CDN URL — both in your own code and inside the sol-query modules. No bundler, no npm install.

Minimal setup

Three things go in <head>: the importmap, a module script that makes rdflib available globally, and the component loader.

Complete minimal page
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">

  <!-- 1. Tell the browser where to find rdflib -->
  <script type="importmap">
    {"imports": {"rdflib": "https://esm.sh/rdflib"}}
  </script>

  <!-- 2. Load rdflib and expose it so sol-query can find it -->
  <script type="module">
    import * as $rdf from 'rdflib';
    window.$rdf = $rdf;
  </script>

  <!-- 3. Load the component -->
  <script type="module" src="./sol-query/sol-query.js"></script>
</head>
<body>

  <sol-query endpoint="https://example.org/data.ttl"></sol-query>

</body>
</html>
Steps 1 and 2 can be combined: the importmap alone is enough if you assign window.$rdf before calling sol-query, which the separate module script ensures via the browser's module execution order.

Query examples

View all statements in a document

<sol-query
  endpoint="https://example.org/data.ttl"
></sol-query>

View all statements about a subject

Include a #fragment in the endpoint — the document is fetched and filtered to triples whose subject matches the full URI.

<sol-query
  endpoint="https://example.org/data.ttl#Alice"
></sol-query>

Mini-query (triple pattern)

Use ? as a wildcard, CURIEs from the built-in prefix table, or full <URIs>. Everything after the predicate that is not a URI or CURIE is treated as a literal.

<!-- All people in the document -->
<sol-query
  endpoint="https://example.org/data.ttl"
  wanted="? rdf:type foaf:Person"
></sol-query>

<!-- Person named "Alice Smith" -->
<sol-query
  endpoint="https://example.org/data.ttl"
  wanted="? foaf:name Alice Smith"
></sol-query>

Inline SPARQL

<sol-query
  endpoint="https://dbpedia.org/sparql"
  sparql="SELECT ?name WHERE {
    <http://dbpedia.org/resource/Tim_Berners-Lee> foaf:name ?name .
    FILTER(lang(?name) = 'en')
  } LIMIT 1"
></sol-query>

Stored SPARQL (query from an RDF file)

<sol-query
  endpoint="https://dbpedia.org/sparql"
  sparql="https://example.org/queries.ttl#FindPeople"
></sol-query>

Replaceable parameters

Use {{name}} placeholders in the query; supply values with var-* attributes. Values are sanitized to prevent SPARQL injection.

<sol-query
  endpoint="https://dbpedia.org/sparql"
  sparql="SELECT ?name WHERE {
    ?p a <http://dbpedia.org/ontology/Person> ;
       dbo:birthPlace <http://dbpedia.org/resource/{{city}}> ;
       foaf:name ?name .
    FILTER(lang(?name) = 'en')
  } LIMIT {{limit}}"
  var-city="London"
  var-limit="10"
></sol-query>

Format variants

The format attribute accepts table (default), dl (definition list using col 1 as term, col 2 as definition), or list (single-column unordered list).

<sol-query endpoint="data.ttl" format="dl"></sol-query>
<sol-query endpoint="data.ttl#Alice" wanted="? foaf:name ?" format="list"></sol-query>

Script API (no component needed)

Use runQuery from rdf-utils.js directly when you want data in JavaScript rather than rendered HTML.

<!DOCTYPE html>
<html>
<head>
  <script type="importmap">
    {"imports": {"rdflib": "https://esm.sh/rdflib"}}
  </script>
</head>
<body>
<script type="module">
  import * as $rdf from 'rdflib';
  window.$rdf = $rdf;

  import { runQuery } from './sol-query/shared/rdf-utils.js';

  // Returns a plain string for 1×1 results, array of objects otherwise
  const name = await runQuery({
    endpoint: 'https://example.org/profile.ttl',
    wanted: '? foaf:name ?'
  });
  console.log(name);
</script>
</body>
</html>

See Script API docs for the full reference.

CORS & hosting

The component files must be served, not opened as files

Because sol-query uses ES modules (type="module"), the HTML page and the sol-query/ directory must be served over HTTP, not opened directly as file:// paths. Any static file server works:

# Python
python3 -m http.server 8080

# Node / npx
npx serve .

# VS Code
# Install the "Live Server" extension and click "Go Live"

Remote RDF documents and SPARQL endpoints

Fetching data from another origin requires that server to send CORS headers. Public endpoints like dbpedia.org/sparql and many Solid servers already do. If you control the data server, add:

Access-Control-Allow-Origin: *
Access-Control-Allow-Headers: Accept
If a remote endpoint does not send CORS headers, the browser will block the request. This is a server-side restriction; it cannot be worked around in the browser. Use a proxy or fetch on the server side instead.

Solid pods (authenticated requests)

Public resources on Solid pods work without authentication. For private resources, integrate @inrupt/solid-client-authn-browser and pass session.fetch to the lower-level loadRdfStore(url, session.fetch) function from rdf-utils.js.

Security

Blocked SPARQL operations

All paths that execute SPARQL — inline, stored, or assembled from parameters — reject queries containing modifying operations:

  • INSERT DATA, INSERT INTO
  • DELETE DATA, DELETE WHERE, DELETE FROM
  • DROP, CREATE, CLEAR, LOAD
  • COPY, MOVE, ADD

A blocked query shows an error message and is never sent to the endpoint.

Variable sanitization

Values supplied via var-* attributes are sanitized before substitution:

  • Brace characters { } are disallowed (prevent SPARQL structural injection)
  • The blocked keywords above are rejected in variable values too
  • Backslash and quote characters are escaped to prevent string-literal breakout
Sanitization is a defence-in-depth measure. Do not use var-* to substitute raw URIs or namespace prefixes into queries — use parameterized queries or fixed URIs in the query text instead.

XSS

The renderer builds the DOM using createElement / textContent; it never uses innerHTML for data values. URI links are created with a.href = value (safe attribute assignment). Data from the RDF store is never eval'd or injected as raw HTML.

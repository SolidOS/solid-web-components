Install

One script tag (no auth)

<script src="https://your-host/dist/solid-web-components.bundle.min.js"></script>

<sol-menu>
  <a href="readme.md">Readme</a>
  <a href="data.ttl" handler="sol-query">Data</a>
</sol-menu>

Every component (sol-menu, sol-include, sol-query, sol-login) is registered and every common dep (rdflib, dompurify, marked, …) is bundled in. No importmap, no npm install, no build step. Drop the file on a static host or use a CDN that mirrors the repo's dist/ folder.

Two script tags (with login)

<script src="https://your-host/dist/vendor/@inrupt-solid-client-authn-browser.umd.js"></script>
<script src="https://your-host/dist/solid-web-components.bundle.min.js"></script>

<sol-login issuers="https://solidcommunity.net,https://login.inrupt.com"></sol-login>
<sol-query endpoint="https://your-pod.example/data.ttl" pattern="?s ?p ?o"></sol-query>

The Solid OIDC auth client is "bring your own" so demos that don't use login don't pay for it. Once <sol-login> is on the page and the user logs in, every <sol-query> / <sol-include> on the page automatically uses the authenticated fetch — Comunica SPARQL queries against authenticated endpoints, document includes from authenticated pods, all of it. No wiring required.

The bundle is ~680 KB minified; the auth client is another ~310 KB and only loads if you include it. If file size matters more, the alternatives are below.

1. One script tag (simplest)

Use dist/solid-web-components.bundle.min.js. Everything is inlined: every component plus every external runtime dep (rdflib, dompurify, marked). No globals to wire up, no order to get right.

<script src="https://your-host/dist/solid-web-components.bundle.min.js"></script>

<sol-include source="readme.md"></sol-include>
<sol-query   endpoint="data.ttl" pattern="?s ?p ?o"></sol-query>
<sol-menu>...</sol-menu>
What's included: sol-menu, sol-include, sol-query, sol-login plus every dep they need at runtime. Auth (the inrupt OIDC client) is bring-your-own — drop in dist/vendor/@inrupt-solid-client-authn-browser.umd.js as a separate <script> tag before the bundle if you need login. Components requiring extra externals (sol-form needs solid-ui; sol-solidos needs mashlib) aren't in this bundle yet — use the importmap path for those.

Auth fetch propagation

When <sol-login> is on the page and the user is logged in, <sol-query> and <sol-include> automatically use the authenticated fetch for every request — including Comunica's SPARQL queries against authenticated endpoints. The lookup is page-wide and light-DOM only; if you have multiple <sol-login> elements (advanced) you can target a specific one with the login attribute (CSS selector):

<sol-login id="primary"></sol-login>
<sol-login id="alt"></sol-login>

<sol-query login="#alt" endpoint="https://other.pod/data.ttl" ...></sol-query>

2. Per-component bundle (smaller)

If you only need one or two components and care about size, load just those. Each component ships its own UMD bundle in dist/. You add one <script> tag per external it needs (loaded as plain UMD globals from a CDN like unpkg).

Zero-dep components — one tag

sol-menu and sol-tabs have no externals at all:

<script src="https://your-host/dist/sol-menu.umd.min.js"></script>
<sol-menu>...</sol-menu>

sol-include — three tags

<script src="https://unpkg.com/dompurify@3/dist/purify.min.js"></script>
<script src="https://unpkg.com/marked@4/marked.min.js"></script>
<script src="https://your-host/dist/sol-include.umd.min.js"></script>

sol-query — four tags

<script src="https://unpkg.com/rdflib@2/dist/rdflib.min.js"></script>
<script src="https://unpkg.com/dompurify@3/dist/purify.min.js"></script>
<script src="https://unpkg.com/marked@4/marked.min.js"></script>
<script src="https://your-host/dist/sol-query.umd.min.js"></script>
ComponentExternals required (load these as <script> tags before the component)
sol-menu, sol-tabs, sol-accordion, sol-rolodex, sol-modalnone
sol-includedompurify, marked
sol-pod, sol-pod-ops, sol-wac, sol-loginrdflib, dompurify (sol-login also: @inrupt/solid-client-authn-browser)
sol-queryrdflib, dompurify, marked (+ @comunica/query-sparql only for remote SPARQL endpoints)
sol-formrdflib, dompurify, marked, n3, rdf-validate-shacl, solid-ui

sol-solidosrdflib, dompurify, marked, mashlib

Externals must come before the component script. They register as globals ($rdf, DOMPurify, marked) which the component bundles look up.

3. With a bundler (Vite, webpack, Rollup, esbuild…)

Building an app with a bundler? Install the package and import what you want. The bundler reads solid-web-components's exports map and resolves the right entry per runtime — browser builds get the web component, Node builds get the Node API.

npm install solid-web-components
import 'solid-web-components/menu';
import 'solid-web-components/include';
import 'solid-web-components/query';
// or all at once:
import 'solid-web-components/web';

Heavy environment-specific deps (jsdom, @comunica/query-sparql, the inrupt auth packages, solid-ui) are optional peer dependencies — your install only pulls what you ask for. Add them as needed:

npm install @comunica/query-sparql       # only if you query remote SPARQL endpoints

4. ES modules + importmap (for source-level loading)

If you want to load the source ES modules directly (no bundler, no UMD), the browser needs an importmap so bare specifiers like rdflib resolve. List only the externals your components use — the full 4-entry map is overkill for one component.

For sol-include:

<script type="importmap">
{
  "imports": {
    "dompurify": "https://esm.sh/dompurify@3",
    "marked":    "https://esm.sh/marked@4"
  }
}
</script>
<script type="module" src="https://your-host/web/sol-include.js"></script>

For sol-full.js (registers all components covered by the importmap):

<script type="importmap">
{
  "imports": {
    "rdflib":                 "https://esm.sh/rdflib@2",
    "dompurify":              "https://esm.sh/dompurify@3",
    "marked":                 "https://esm.sh/marked@4",
    "@comunica/query-sparql": "https://esm.sh/@comunica/query-sparql@5"
  }
}
</script>
<script type="module" src="https://your-host/web/sol-full.js"></script>
Why inline the JSON? External-file importmaps (<script type="importmap" src="…">) work only in Chrome 124+, Firefox 132+, Safari 16.4+. Inline JSON works everywhere.

5. Offline / vendored (no CDN at runtime)

For air-gapped deployments. The single-tag bundle from option 1 is already self-contained — host it locally and you're done. If you prefer source ES modules instead, vendor every external as a local file:

git clone <repo-url>
cd solid-web-components
npm install
npm run build      # generates dist/importmap-{cdn,local}.json + dist/vendor/*.js + bundles

Serve the working tree with any static HTTP server, then use the local importmap:

<script type="importmap">
{
  "imports": {
    "rdflib":                 "../dist/vendor/rdflib.js",
    "dompurify":              "../dist/vendor/dompurify.js",
    "marked":                 "../dist/vendor/marked.js",
    "@comunica/query-sparql": "../dist/vendor/@comunica-query-sparql.js"
  }
}
</script>
<script type="module" src="../web/sol-full.js"></script>

Verify with test-offline.html — DevTools → Network → Offline; every check should be green and zero esm.sh/unpkg/jsdelivr URLs should appear.

Build commands (only needed to regenerate dist/)

ScriptWhat it does
npm run bundlePer-component UMDs (dist/sol-*.umd.min.js) plus the all-in-one dist/solid-web-components.bundle.min.js.
npm run build:importmapsReads tools/external-deps.json; writes dist/importmap-cdn.json and dist/importmap-local.json.
npm run build:vendorFor each external dep, runs Rollup once to produce dist/vendor/<name>.js — a single self-contained ESM file. Idempotent. --force rebuilds everything.
npm run buildAll of the above.

Troubleshooting

"Custom element 'sol-query' is not defined"

The component script hasn't loaded yet — confirm the <script> URL resolves (open it in the browser; you should see minified JS, not a 404). The all-in-one bundle and the per-component UMDs both register their elements as soon as the script runs.

"$rdf is not defined" (per-component mode)

You loaded a component that needs rdflib without first loading rdflib. Put externals before the component:

<script src=".../rdflib.min.js"></script>     <!-- first -->
<script src=".../sol-query.umd.min.js"></script> <!-- after -->

"Failed to resolve module specifier 'rdflib'" (importmap mode)

  1. The importmap is missing the entry — add it, or switch to the all-in-one bundle.
  2. The importmap is loaded after the first <script type="module"> — order matters: 
    <script type="importmap">{ ... }</script>   <!-- always first -->
<script type="module"    src="..."></script>
  3. You used the external-file form in an older browser — inline the JSON.

404 on a vendored file in offline mode

Either npm run build:vendor hasn't run, or the importmap's relative path doesn't match where dist/vendor/ sits relative to the page. Importmap paths are resolved against the importmap's own URL, not the page URL.

sol-login — Script API

Use AuthManager directly from JavaScript for full control over multi-session authentication, authenticated fetch, and login flows.

Setup

Prerequisites

@inrupt/solid-client-authn-browser must be available as a UMD global at window.solidClientAuthn.Session before importing sol-login.js.

UMD bundle (simplest)
<script src="solid-client-authn-browser.bundle.js"></script>
<script type="module">
  import { AuthManager } from './sol-login.js';
</script>
From npm (e.g. in a bundled app)
import { Session } from '@inrupt/solid-client-authn-browser';
window.solidClientAuthn = { Session };

import { AuthManager } from './sol-login.js';
The Session class must be set on window.solidClientAuthn before any AuthManager method creates a session.

fetchFor

const authFetch = auth.fetchFor(url, tag?); const response = await authFetch(url, { method: 'GET' });

fetchFor returns a fetch function, not a response. Call the returned function like you would call fetch().

When called without a tag, it searches all sessions for one whose issuer or WebID covers the URL's origin. When called with a tag, it uses that tag's session.

If no session covers the URL, fetchFor returns the global fetch — so unauthenticated requests always work.

Sessions & Cross-Origin Coverage

Sessions are identified by a session ID derived from the tag and origin: sol_<tag>_<origin>. This means each tag can have an independent session per origin.

A session "covers" an origin if:

  • The session's issuer or WebID host matches the target host
  • The target host is a subdomain of the issuer/WebID host
  • They share the same base domain (e.g. pod.example.com and login.example.com)

This allows a single login to work across related services on the same domain.

Manual Initialization

When used inside a larger app (e.g. Podz), you typically want to control when initialization happens:

const login = document.createElement('sol-login');
login._manualInit = true;  // prevent auto-init on DOMContentLoaded
document.body.appendChild(login);

// Set up sessions with custom tags
const auth = login.auth;

// Later, after your app is ready:
await login.initialize(['left', 'right']);

// Now use auth
const resp = await auth.fetchFor(url, 'left')(url);
initialize() calls handleIncomingRedirect() for all tags and fires the sol-login event if a session was restored from a redirect.

Examples

Read a private file

const auth = new AuthManager();
auth.sessionFor('default');
await auth.handleIncomingRedirect();

const url = 'https://pod.example/private/data.ttl';
const authFetch = auth.fetchFor(url, 'default');
const resp = await authFetch(url);
const text = await resp.text();

Write to a pod

const authFetch = auth.fetchFor(url, 'default');
await authFetch(url, {
  method: 'PUT',
  headers: { 'Content-Type': 'text/turtle' },
  body: '@prefix foaf: <http://xmlns.com/foaf/0.1/> .\n<> a foaf:Person .'
});

Skip auth for public data

auth.noAuth = ['https://dbpedia.org', 'https://www.wikidata.org'];

// These will use plain fetch, no OIDC headers
const resp = await auth.fetchFor('https://dbpedia.org/data/Berlin.ttl')('https://dbpedia.org/data/Berlin.ttl');

Multi-session (two panels)

auth.setSideOrigin('left',  'https://pod-a.example/');
auth.setSideOrigin('right', 'https://pod-b.example/');

const fetchA = auth.fetchFor('https://pod-a.example/file.ttl', 'left');
const fetchB = auth.fetchFor('https://pod-b.example/file.ttl', 'right');

Trigger login if needed

const authenticated = await auth.ensureAuthenticated('https://pod.example/', 'default');
if (!authenticated) {
  // Page is redirecting to the OIDC provider — save state first
}

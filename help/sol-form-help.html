sol-form Component

A generic RDF form renderer. Point it at any Turtle file containing a ui:Form definition and it renders an interactive form. Data lives in an rdflib store; on save, the RDF is written to a Solid Pod or downloaded as a local Turtle file.

Attributes

AttributeDescription
sourceRequired. URI of a Turtle file containing a ui:Form definition. The form's fields, labels, types, and structure are all read from this file.
subjectURI of an existing RDF resource to edit. When set, the component loads the resource and populates the form with existing values. When omitted, a blank node is created for new data entry.
shapeURI of a SHACL shapes file. When set, the form data is validated against the shape before saving. Validation errors are displayed inline and block the save.
save-toPre-fills the Pod URL input for the "Save to Pod" option.

Events

EventDetailWhen
sol-form-change{ subject, ok, message }Fired on every field edit.
sol-form-save{ subject, turtle, target }Fired after a successful save. target is the Pod URL or "local".

Basic demo

This demo loads the contact form definition from data/contact-form.ttl. Fill in fields and use "Show Turtle" to see the RDF output, or use the save bar to download.

<sol-form source="../data/contact-form.ttl"></sol-form>

Writing a form definition

<sol-form> reads standard ui: form vocabulary. A form is a ui:Form with ui:parts listing its fields. Each field has a type (e.g. ui:SingleLineTextField), a ui:label, and a ui:property binding it to an RDF predicate.

When solid-ui is loaded on the page, the full range of field types is available (including ui:Multiple, ui:Options, ui:Choice, ui:ColorField, ui:DateField, etc.). Without solid-ui, a built-in fallback renderer handles common types.

Example form definition:

@prefix ui: <http://www.w3.org/ns/ui#> . @prefix ex: <http://example.org/> . @prefix vcard: <http://www.w3.org/2006/vcard/ns#> . ex:contactForm a ui:Form ; ui:parts ( ex:nameField ex:emailField ex:noteField ) . ex:nameField a ui:SingleLineTextField ; ui:label "Full Name" ; ui:property vcard:fn ; ui:required true . ex:emailField a ui:EmailField ; ui:label "Email" ; ui:property vcard:hasEmail ; ui:required true . ex:noteField a ui:MultiLineTextField ; ui:label "Notes" ; ui:property vcard:note .

Edit mode

Set the subject attribute to load an existing RDF resource for editing. The component fetches the resource, populates the form with current values, and any changes update the in-memory store.

<sol-form
  source="my-form.ttl"
  subject="https://you.pod/data/thing.ttl#it"
></sol-form>
Edit mode requires the subject URL to be fetchable. For Solid Pods, ensure the user is logged in (e.g. via <sol-login>) before the form loads.

SHACL validation

Provide a shape attribute pointing to a SHACL shapes file. On save, the form data is validated against the shape. If validation fails, errors are displayed above the save bar and the save is blocked.

<sol-form
  source="menu-form.ttl"
  shape="shapes/menu.shacl"
></sol-form>

Validation uses rdf-validate-shacl (lazy-loaded). Each violation shows the property path and message from the SHACL shape.

SHACL validation is optional. Without a shape attribute the form saves without validation.

Saving

The save bar at the bottom offers two modes (persisted in localStorage):

ModeHow it works
Save to PodPUTs the serialized Turtle to the specified Pod URL using rdflib's fetcher.webOperation. The user must be authenticated (e.g. via <sol-login>). Pre-fill the URL with the save-to attribute.
Download locallySerializes the store to Turtle and triggers a browser file download. The filename is derived from the ui:label of the form subject.

After a successful save, a sol-form-save event fires with the Turtle content and target in its detail.

Minimal page setup

<!DOCTYPE html>
<html>
<head>
  <script type="importmap">{"imports":{
    "rdflib": "https://esm.sh/rdflib"
  }}</script>
  <script type="module">
    import * as $rdf from 'rdflib';
    window.$rdf = $rdf;
  </script>
  <script type="module" src="sol-form.js"></script>
</head>
<body>
  <sol-form source="my-form.ttl"></sol-form>
</body>
</html>
solid-ui integration: For the full field set (ui:Multiple, ui:Options, etc.), load solid-ui on the page before the form renders. Without it, the built-in fallback renderer handles basic field types.
